# Scrum Helper SPEC

## 1. 프로젝트 개요

Scrum Helper는 소규모 팀이 스크럼 프로젝트를 진행할 때 필요한 팀 관리, 할 일 배정, 진행 확인, 회고 기록을 한곳에서 처리하는 웹 서비스다.

이 문서는 Scrum Helper의 단일 source of truth다. DB 스키마, API 문서, 화면 설계, 구현 계획은 이 문서를 기준으로 파생한다.

### 1.1 목표

- 팀을 만들고 팀원들이 쉽게 참여할 수 있게 한다.
- 팀장이 팀원과 팀 권한을 관리할 수 있게 한다.
- 공개 가입, 비밀번호 가입, 초대코드 가입을 모두 지원한다.
- 팀 단위 task를 만들고 담당자를 복수로 지정할 수 있게 한다.
- task의 완료 여부, 중요도, 마감일을 한눈에 확인할 수 있게 한다.
- task 댓글로 진행 중 의견을 남길 수 있게 한다.
- 팀별 회의록을 여러 개 작성하고 요약을 저장할 수 있게 한다.
- 유저별 회고록을 여러 개 작성하고, 필요한 경우 공동 작업자를 지정해 함께 수정할 수 있게 한다.

### 1.2 대상 사용자

| 사용자 | 설명 |
|---|---|
| 일반 유저 | 회원가입/로그인 후 팀을 생성하거나 기존 팀에 가입하는 사용자 |
| 팀장 | 팀을 생성한 최초 사용자이며, 팀원 관리와 팀장 위임 권한을 가진 사용자 |
| 팀원 | 팀에 가입해 task와 회고록을 작성하는 사용자 |

## 2. 제외 범위

MVP에서는 아래 기능을 구현하지 않는다.

- 회원 탈퇴
- 소셜 로그인
- 이메일 인증
- 알림
- 파일 첨부
- 실시간 동시 편집
- task 상태 세분화: 진행중, 막힘 등
- 스프린트 자동 생성
- 운영 배포 자동화
- 회의록 음성 업로드/요약
- AI 기반 스펙 문서 및 task 자동 생성
- 리더보드, 업적, 성장 나무 같은 게이미피케이션

## 4. 사용자와 권한

### 4.1 회원

회원가입 시 입력값:

- `name`
- `email`
- `password`

정책:

- `email`은 로그인 식별자이며 unique하다.
- `name`은 화면 표시 이름이다.
- DB 내부 관계 연결을 위해 `user_id` 같은 내부 PK는 유지한다.
- 비밀번호는 해시 처리해 저장한다.
- 비밀번호 길이/문자 조합 제약은 MVP에서 두지 않는다.
- 회원 탈퇴는 고려하지 않는다.

### 4.2 팀 권한

팀에는 팀장이 정확히 1명만 존재한다.

| 역할 | 권한 |
|---|---|
| LEADER | 팀 정보 관리, 팀원 제거, 팀장 변경, task 관리, 회고록 작성/수정 |
| MEMBER | task 관리, 댓글 작성, 본인 또는 협업 회고록 작성/수정 |

정책:

- 팀 생성자가 최초 팀장이다.
- 팀장 변경 시 새 팀원이 `LEADER`가 되고 기존 팀장은 `MEMBER`가 된다.
- 팀장은 팀원을 제거할 수 있다.
- 팀장은 본인을 제거할 수 없다.
- 팀에는 항상 1명의 팀장이 유지되어야 한다.
- 팀원 제거 후에도 해당 사용자가 작성한 task, 댓글, 회고록은 유지한다.
- 제거 대상 팀원이 어떤 task의 유일한 담당자라면 팀원 제거 전에 담당자를 재배정해야 한다.
- 제거 대상 팀원이 회고록 공동 작업자였다면 공동 작업자 권한은 제거된다.

## 5. 핵심 기능

### 5.1 인증

| 기능 | 설명 |
|---|---|
| 회원가입 | `name`, `email`, `password`로 가입 |
| 로그인 | `email`, `password`로 로그인 |
| 로그아웃 | 현재 세션 종료 |
| 내 정보 조회 | 로그인한 사용자 정보 확인 |

### 5.2 팀

| 기능 | 설명 |
|---|---|
| 전체 팀 목록 조회 | 로그인한 유저는 모든 팀 목록을 볼 수 있다 |
| 팀 생성 | 팀명, 설명, 선택 비밀번호로 팀 생성 |
| 팀 가입 | 공개 팀은 즉시 가입, 비밀번호 팀은 비밀번호 입력 후 가입 |
| 초대코드 가입 | 팀 초대코드를 입력해 비밀번호와 무관하게 가입 |
| 팀원 목록 조회 | 팀 소속 사용자 목록 조회 |
| 팀원 제거 | 팀장이 팀원을 팀에서 제거 |
| 팀장 변경 | 팀장이 다른 팀원에게 팀장을 위임 |

정책:

- 팀 이름은 unique하다.
- 팀 비밀번호는 선택값이다.
- 팀 비밀번호가 없으면 공개 팀이다.
- 팀 비밀번호가 있으면 비밀번호를 입력해야 가입할 수 있다.
- 팀 비밀번호도 해시 처리해 저장한다.
- 팀 생성 시 unique 초대코드를 자동 발급한다.
- 초대코드를 아는 사용자는 공개/비밀번호 팀 여부와 무관하게 가입할 수 있다.
- 초대코드는 팀장만 재발급할 수 있다.
- 초대코드 재발급 후 기존 초대코드는 사용할 수 없다.

### 5.3 Task

| 기능 | 설명 |
|---|---|
| task 생성 | 제목, 설명, 중요도, 마감일, 담당자를 입력해 생성 |
| task 목록 조회 | 팀의 task 목록 조회 |
| task 수정 | 제목, 설명, 중요도, 마감일, 담당자 수정 |
| task 완료 처리 | 완료/미완료 상태 변경 |
| task 삭제 | 팀원 모두 삭제 가능 |
| 댓글 작성 | task에 의견 댓글 작성 |

정책:

- task 상태는 `미완성`, `완성` 2개만 사용한다.
- task 중요도는 `낮음`, `보통`, `높음` 3단계다.
- task에는 마감일이 필요하다.
- task 담당자는 여러 명 선택할 수 있다.
- task 담당자는 1명 이상 필수다.
- task 담당자는 같은 팀의 팀원만 선택할 수 있다.
- task 완료 처리는 팀원 모두 가능하다.
- task에는 여러 개의 댓글을 달 수 있다.

### 5.4 회의록

팀에는 여러 개의 회의록이 존재할 수 있다.

| 기능 | 설명 |
|---|---|
| 회의록 생성 | 제목, 회의 일시, 회의 원문, 요약을 입력해 생성 |
| 회의록 목록 조회 | 팀에 속한 회의록 목록 조회 |
| 회의록 상세 조회 | 회의 원문과 요약 확인 |
| 회의록 수정 | 작성자 또는 팀장이 수정 가능 |
| 회의록 삭제 | 작성자 또는 팀장이 삭제 가능 |

정책:

- 회의록 작성자는 1명이다.
- 팀원은 회의록을 조회할 수 있다.
- 회의록 수정/삭제는 작성자 또는 팀장만 가능하다.
- 회의록은 이후 스펙 문서 자동 생성의 입력 데이터로 사용한다.

### 5.5 회고록

하나의 스크럼 프로젝트, 즉 하나의 팀에는 여러 개의 회고록이 존재할 수 있다.

회고록은 유저별로 작성할 수 있으며, 필요한 경우 공동 작업자를 선택할 수 있다.

| 기능 | 설명 |
|---|---|
| 회고록 생성 | 제목과 회고 내용을 입력해 생성 |
| 회고록 목록 조회 | 팀에 속한 회고록 목록 조회 |
| 회고록 상세 조회 | 회고록 내용과 작성자/공동 작업자 확인 |
| 회고록 수정 | 작성자와 공동 작업자가 수정 가능 |
| 회고록 삭제 | 작성자와 공동 작업자가 삭제 가능 |
| 공동 작업자 지정 | 같은 팀원을 공동 작업자로 지정 |

회고록 항목:

| 항목 | 설명 |
|---|---|
| 어제 한 일 | 직전 작업 내용 |
| 오늘 할 일 | 당일 계획 |
| 궁금한/필요한/알아낸 것 | 질문, 필요한 도움, 새로 알게 된 내용 |

정책:

- 회고록 작성자는 1명이다.
- 공동 작업자는 여러 명 선택할 수 있다.
- 공동 작업자는 회고록을 수정할 수 있다.
- 공동 작업자는 회고록을 삭제할 수 있다.
- 회고록 본문은 작성자와 공동 작업자가 수정할 수 있다.
- 회고록 공동 작업자 목록 변경은 작성자만 가능하다.
- 작성자와 공동 작업자는 모두 화면에 표시한다.
- 회고록은 날짜별 1개로 제한하지 않는다.

## 6. 엔티티 초안

이 섹션은 DB 스키마가 아니라 도메인 엔티티 초안이다. 실제 DB 타입, PK/FK, NN 제약은 다음 단계에서 확정한다.

### 6.1 User

| 속성 | 설명 | 제약 |
|---|---|---|
| id | 내부 사용자 ID | PK |
| name | 화면 표시 이름 | required |
| email | 로그인 이메일 | required, unique |
| passwordHash | 해시된 비밀번호 | required |
| createdAt | 생성 시각 | required |
| updatedAt | 수정 시각 | required |

### 6.2 Team

| 속성 | 설명 | 제약 |
|---|---|---|
| id | 내부 팀 ID | PK |
| name | 팀 이름 | required, unique |
| description | 팀 설명 | optional |
| passwordHash | 팀 가입 비밀번호 해시 | optional |
| inviteCode | 팀 가입 초대코드 | optional, unique |
| leaderUserId | 현재 팀장 ID | required, FK User |
| createdAt | 생성 시각 | required |
| updatedAt | 수정 시각 | required |

### 6.3 TeamMember

| 속성 | 설명 | 제약 |
|---|---|---|
| id | 멤버십 ID | PK |
| teamId | 팀 ID | required, FK Team |
| userId | 사용자 ID | required, FK User |
| role | 팀 역할 | `LEADER` 또는 `MEMBER` |
| joinedAt | 가입 시각 | required |

제약:

- `(teamId, userId)`는 unique하다.
- 한 팀에는 `LEADER`가 정확히 1명이어야 한다.

### 6.4 Task

| 속성 | 설명 | 제약 |
|---|---|---|
| id | task ID | PK |
| teamId | 팀 ID | required, FK Team |
| title | task 제목 | required |
| description | task 설명 | optional |
| priority | 중요도 | `LOW`, `MEDIUM`, `HIGH` |
| dueDate | 마감일 | required |
| completed | 완료 여부 | boolean |
| createdByUserId | 생성자 | required, FK User |
| createdAt | 생성 시각 | required |
| updatedAt | 수정 시각 | required |

### 6.5 TaskAssignee

| 속성 | 설명 | 제약 |
|---|---|---|
| taskId | task ID | PK, FK Task |
| userId | 담당자 ID | PK, FK User |
| assignedAt | 배정 시각 | required |

제약:

- `(taskId, userId)` 복합 PK를 사용한다.
- 담당자는 해당 task가 속한 팀의 팀원이어야 한다.

### 6.6 TaskComment

| 속성 | 설명 | 제약 |
|---|---|---|
| id | 댓글 ID | PK |
| taskId | task ID | required, FK Task |
| userId | 작성자 ID | required, FK User |
| content | 댓글 내용 | required |
| createdAt | 생성 시각 | required |
| updatedAt | 수정 시각 | required |

### 6.7 Retrospective

| 속성 | 설명 | 제약 |
|---|---|---|
| id | 회고록 ID | PK |
| teamId | 팀 ID | required, FK Team |
| authorUserId | 작성자 ID | required, FK User |
| title | 회고록 제목 | required |
| yesterdayWork | 어제 한 일 | optional |
| todayPlan | 오늘 할 일 | optional |
| note | 궁금한/필요한/알아낸 것 | optional |
| createdAt | 생성 시각 | required |
| updatedAt | 수정 시각 | required |

### 6.8 RetrospectiveCollaborator

| 속성 | 설명 | 제약 |
|---|---|---|
| retrospectiveId | 회고록 ID | PK, FK Retrospective |
| userId | 공동 작업자 ID | PK, FK User |
| addedAt | 추가 시각 | required |

제약:

- 공동 작업자는 해당 회고록이 속한 팀의 팀원이어야 한다.
- 작성자는 공동 작업자 목록에 중복 저장하지 않는다.

## 7. 주요 시나리오

### 7.1 팀 생성과 가입

1. 사용자가 회원가입 또는 로그인한다.
2. 전체 팀 목록을 확인한다.
3. 원하는 팀이 없으면 새 팀을 생성한다.
4. 팀 생성자는 자동으로 팀장이 된다.
5. 다른 유저는 전체 팀 목록에서 해당 팀을 선택한다.
6. 공개 팀이면 별도 확인 없이 즉시 가입한다.
7. 비밀번호 팀이면 팀 비밀번호를 입력해 가입한다.
8. 초대코드를 받은 유저는 팀 목록 화면에서 초대코드를 입력해 가입한다.
9. 가입 후 팀 대시보드로 이동한다.

### 7.2 팀장 변경

1. 팀장이 팀원 목록을 확인한다.
2. 새 팀장으로 지정할 팀원을 선택한다.
3. 시스템은 기존 팀장을 `MEMBER`로 변경한다.
4. 시스템은 선택된 팀원을 `LEADER`로 변경한다.
5. 팀에는 여전히 팀장이 1명만 존재한다.

### 7.3 Task 배정

1. 팀원이 task를 생성한다.
2. 중요도와 마감일을 입력한다.
3. 담당자를 1명 이상 또는 여러 명 선택한다.
4. 시스템은 같은 팀의 팀원만 담당자로 저장한다.
5. 담당자와 팀원은 task 목록에서 task를 확인한다.

### 7.4 Task 완료와 댓글

1. 팀원이 task 상세 화면에 들어간다.
2. 의견이 있으면 댓글을 작성한다.
3. task가 끝나면 완료 상태로 변경한다.
4. task 목록에서 완료/미완료가 구분되어 보인다.

### 7.5 회고록 작성

1. 팀원이 회고록을 생성한다.
2. `어제 한 일`, `오늘 할 일`, `궁금한/필요한/알아낸 것`을 작성한다.
3. 필요한 경우 공동 작업자를 선택한다.
4. 작성자와 공동 작업자는 회고록을 수정할 수 있다.
5. 공동 작업자 목록은 작성자만 변경할 수 있다.
6. 팀 회고록 목록에서 여러 회고록을 확인할 수 있다.

## 8. 화면 구조 초안

| 화면 | 주요 기능 |
|---|---|
| 로그인/회원가입 | 계정 생성, 로그인 |
| 팀 목록 | 전체 팀 목록, 공개/비밀번호 팀 구분, 팀 가입, 초대코드 가입, 팀 생성 |
| 팀 상세 대시보드 | 팀 정보, 팀원 요약, task 요약, 회고록 요약 |
| 팀원 관리 | 팀원 목록, 팀원 제거, 팀장 변경 |
| Task 목록 | 완료/미완료 task, 중요도, 마감일, 담당자 표시 |
| Task 상세 | task 수정, 완료 처리, 담당자 변경, 댓글 |
| 회고록 목록 | 팀 회고록 목록, 작성자, 공동 작업자 표시 |
| 회고록 상세 | 회고 내용 조회/수정, 공동 작업자 관리 |

## 9. 확정된 기술/권한 결정

| 항목 | 결정 |
|---|---|
| 프론트엔드 프레임워크 | React |
| DB | MySQL |
| ORM | Spring Data JPA |
| 인증 방식 | JWT |
| task 삭제 권한 | 팀원 모두 |
| 댓글 수정/삭제 권한 | 작성자만 |
| 회고록 삭제 권한 | 작성자 + 공동작업자 |
| 팀 비밀번호 변경 | MVP 포함 |
| 팀 설명 수정 | 팀장만 |
| 팀 가입 후 기본 화면 | 팀 대시보드 |
| task 담당자 최소 인원 | 1명 이상 필수 |
| task 완료 처리 권한 | 팀원 모두 |
| 회고록 공동 작업자 변경 권한 | 작성자만 |
| 팀원 제거 시 기존 task/회고록 처리 | 작성 기록 유지 |
| 팀원 제거 시 task 담당자 처리 | 유일 담당자인 task가 있으면 제거 전 재배정 필요 |
| 팀원 제거 시 회고록 공동 작업자 처리 | 공동 작업자 권한 제거 |
| 비밀번호 없는 공개 팀 가입 | 즉시 가입 |
| 초대코드 가입 | 공개/비밀번호 팀 모두 즉시 가입 |
