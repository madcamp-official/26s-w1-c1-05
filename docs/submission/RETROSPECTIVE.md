# Scrum Helper Retrospective

## 1. 역할 분담

| 팀원 | 담당 |
|---|---|
| 안종화 | 백엔드 API, DB 설계, 인증/권한, Gemini API, 배포, 테스트 데이터 |
| 김희서 | 프론트엔드 화면, 사용자 흐름, 디자인 시스템 적용, 시연 UX |

## 2. 개발 과정에서 어려웠던 점

### 2.1 프론트엔드와 백엔드 병합

각자 다른 속도로 화면과 API를 구현하면서 merge 이후 타입 오류, API 응답 구조 불일치, 일부 route 불일치가 발생했다.

해결 방법:

- 백엔드는 기존 API 정책과 데이터 무결성을 우선 유지했다.
- 프론트는 팀원이 구현한 화면 구조를 최대한 유지했다.
- API 문서를 기준으로 request/response mapping을 다시 정리했다.
- `npm run build`, `./gradlew test`, 배포 서버 smoke test로 회귀를 확인했다.

### 2.2 AI 기능의 안정성

Gemini API는 네트워크, API key, 응답 format 문제로 항상 성공한다고 가정하기 어렵다.

해결 방법:

- Gemini API 성공 시 `generatedBy = GEMINI`으로 응답한다.
- 실패하거나 키가 없으면 `LOCAL_FALLBACK` 결과를 반환한다.
- Spec 생성은 선택한 회의록 내용만 근거로 하도록 프롬프트를 조정했다.
- Todo prompt는 AI 티가 덜 나고 실제 작업자가 바로 사용할 수 있는 브리프 형태로 튜닝했다.

### 2.3 DB 상태와 발표 데이터

발표 시연은 항상 같은 데이터로 시작해야 하지만, 실제 시연 중에는 가입, Task 생성, 상태 변경으로 DB가 계속 바뀐다.

해결 방법:

- Team A는 완료된 프로젝트, Team B는 진행 중 프로젝트로 baseline을 분리했다.
- 발표 전 DB reset을 통해 동일한 상태로 복원할 수 있게 했다.
- Demo 계정, 팀 비밀번호, 초대코드를 문서화했다.

### 2.4 권한 정책

팀장, 팀원, 작성자, 공동 작업자 권한이 기능마다 달라 실수하기 쉬웠다.

해결 방법:

- 팀 접근은 `NOT_TEAM_MEMBER`로 일관되게 차단했다.
- 회고록 공동 작업자 목록은 작성자만 바꿀 수 있게 했다.
- 댓글 수정/삭제는 작성자만 가능하게 제한했다.
- 팀장 변경은 transaction으로 단일 팀장 invariant를 유지했다.

## 3. 구현하면서 배운 점

- 문서가 먼저 정리되어 있어야 프론트와 백엔드 작업이 충돌해도 빠르게 기준을 잡을 수 있다.
- AI 기능은 API 호출 성공보다 실패 시 UX를 먼저 설계해야 실제 서비스처럼 보인다.
- 배포 서버와 로컬 서버는 환경변수, DB 계정, CORS, 도메인 설정을 명확히 분리해야 한다.
- 발표용 데이터는 기능 테스트 데이터와 달리 스토리 흐름을 기준으로 만들어야 한다.

## 4. 아쉬운 점

- 실시간 협업, 알림, 이메일 전송까지 완성하지는 못했다.
- 모바일 화면을 더 깊게 최적화할 시간이 부족했다.
- AI 응답 품질을 더 많은 실제 회의록으로 검증하지 못했다.
- DB migration 도구를 정식으로 도입하지 못하고 JPA/SQL 기반으로 관리했다.
